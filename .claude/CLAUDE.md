# Konsensus — Инструкции для ИИ-ассистента

## Проект

**Konsensus** — платформа для разрешения споров с ИИ-медиатором. Пользователи создают спор, обмениваются аргументами, а ИИ анализирует позиции и предлагает решения.

## Стек технологий

- **Frontend:** Next.js 15 (App Router) + TypeScript + Tailwind CSS
- **Backend/DB:** Supabase (PostgreSQL + Auth + Realtime)
- **ИИ:** Groq API (llama-3.3-70b-versatile)
- **Деплой:** Vercel (автодеплой из main)

## Языки

- **Код:** английский (переменные, функции, комментарии в коде)
- **Документация:** русский (docs/, README, CLAUDE.md, логи)
- **UI:** русский (пользовательский интерфейс)

## Правила кода

- TypeScript strict mode (`strict: true` в tsconfig)
- ESLint — следовать конфигу из Next.js
- Naming conventions:
  - Компоненты: PascalCase (`DisputeCard.tsx`)
  - Утилиты/хуки: camelCase (`useDispute.ts`, `formatDate.ts`)
  - Типы/интерфейсы: PascalCase с префиксом (`type DisputeStatus`, `interface Dispute`)
  - Константы: UPPER_SNAKE_CASE
  - Файлы в `src/app/`: kebab-case (Next.js convention)
- Компоненты: функциональные, с хуками
- Импорты: использовать alias `@/*` для `src/*`

## Workflow-правила

### Логирование
- Логировать каждый milestone в `docs/log/YYYY-MM-DD.md`
- Формат: что сделано, решения, следующие шаги, заметки
- Использовать skill `/log` для быстрого добавления записей

### Коммиты
- Коммитить после каждого завершённого этапа
- Сообщения коммитов на английском, краткие и осмысленные
- Напоминание: пушить после каждых 3-5 коммитов

### Безопасность
- Перед коммитом — ВСЕГДА проверить что `.env*` файлы НЕ попадут в git
- Секреты — только через переменные окружения
- Шаблон переменных: `.env.example` (без реальных значений)

### Обновление статуса проекта
- При завершении milestone — обновить `docs/status.md`:
  - Отметить выполненные пункты чеклиста
  - Обновить таблицу технологий (если добавлены новые)
  - Обновить таблицу веток (если созданы новые)
  - Обновить таблицу workflow/инструментов
  - Добавить запись в "История изменений"
- Также обновить секцию "Текущий статус" в этом CLAUDE.md
- Мелкие изменения — только по команде `/update-instructions`

## Поведение при старте сессии

Когда пользователь начинает новую сессию или просит продолжить работу:
1. Прочитать текущий статус в этом файле (секция "Текущий статус")
2. Проверить последний лог в `docs/log/`
3. Посмотреть `docs/roadmap.md` для понимания следующих шагов
4. Предложить конкретный план действий на эту сессию
5. Спросить пользователя, согласен ли он с планом
