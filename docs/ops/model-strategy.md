# Стратегия по моделям и стоимости

## Базовая гипотеза

Не строить продукт на предположении, что появится удобный "безлимитный API по подписке".
Архитектура Konsensus должна исходить из metered inference и управляемой маршрутизации запросов.

## Что это значит для продукта

- дешёвые и быстрые задачи должны жить на дешёвой/быстрой модели
- дорогие reasoning-задачи должны вызываться только там, где они реально влияют на качество результата
- смена провайдера не должна требовать переписывания продуктовой логики

## Целевое разбиение по типам задач

### Лёгкий слой
- категоризация темы
- waiting-insights
- короткие public summaries
- редкие аренные AI-комментарии

Требования:
- низкая задержка
- дешёвый токен
- терпимость к краткому ответу

### Тяжёлый слой
- финальная медиация
- high-heat кейсы
- глубокий анализ многоходового спора
- будущий premium / show / orchestrator flow

Требования:
- лучшее reasoning-качество
- больший контекст
- более строгий контроль структуры ответа

## Архитектурный следующий шаг

Перед реальной сменой провайдера добавить конфиг-реестр вида:

- `topic_categorization`
- `waiting_insight`
- `round_public_summary`
- `round_private_insight`
- `final_mediation`
- `arena_comment`

Для каждой фичи должно быть видно:
- какая модель используется сейчас
- какой провайдер её отдаёт
- почему именно она выбрана

Следующий продуктовый слой для AI описан отдельно в
[ai-orchestration-rollout.md](/C:/project21/konsensus/docs/ops/ai-orchestration-rollout.md):
- сначала role-based orchestration внутри текущего AI-слоя;
- потом feature registry;
- потом логирование agent runs;
- и только потом real multi-model routing.

## Как принимать решение о будущем провайдере

Сравнивать не только "цену за миллион токенов", но и:
- задержку
- стабильность structured output
- качество русского языка
- качество reasoning именно на конфликтных сценариях
- удобство API и ограничения по rate limit

## Практическая позиция на сейчас

- текущий дешёвый слой можно продолжать держать на fast/cheap модели
- более сильную модель подключать позже точечно, а не переводить на неё весь продукт целиком
- если появится новый провайдер, интеграция должна происходить через feature-based routing, а не через массовую замену везде
- orchestration rollout должен встраиваться в эту модель, а не обходить её
