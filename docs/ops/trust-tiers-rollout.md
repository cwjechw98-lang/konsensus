# Trust Tiers Rollout

## Цель

Ввести типы аккаунтов и уровни доверия, чтобы публичный слой Konsensus не деградировал в спам, накрутки и шум.

## Зачем это нужно

- публичные споры без trust model масштабируются плохо;
- не всем аккаунтам нужен одинаковый доступ;
- тяжёлая KYC-модель на старте слишком дорогая и мешает росту.

## Решение v1

Три уровня:
- `basic`
- `linked`
- `trusted`

### Basic
- может участвовать в приватных спорах;
- может читать публичные;
- не может инициировать публичные споры;
- получает более строгие лимиты.

### Linked
- аккаунт с более надёжным сигналом входа и историей;
- может комментировать и участвовать в публичных форматах с лимитами;
- может быть допущен к более широкому social layer.

### Trusted
- может инициировать публичные споры;
- может иметь расширенные публичные права;
- приоритетнее в социально значимых сценариях.

## Источники сигнала для v1

Без документов на первом этапе:
- способ входа;
- возраст аккаунта;
- история споров;
- история moderation flags;
- базовые reputation signals.

## Встраивание в текущий код

### v1
- добавить поле trust tier в профиль или отдельную таблицу user_access_state
- rule-based evaluator
- публичные операции проверяют tier:
  - public dispute creation
  - public comments
  - отдельные social features

## Что пока не делать

- документную верификацию
- платный доступ как единственный путь к high trust
- сложную антифрод-систему с внешними провайдерами

## Что нужно добить после v1

### v1.1
- автоматический пересчёт tier по поведенческим сигналам
- audit trail изменения tier

### v1.2
- связать tier с reputation и appeals
- подготовить optional document verification path

### v2
- high-trust verification для публично значимых сценариев
- расширенные community / moderation права
- связка trust tiers с B2B и экспертными ролями
