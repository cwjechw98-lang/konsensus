# Navigation + Onboarding Rollout

## Цель

Пересобрать product shell Konsensus так, чтобы:
- залогиненный пользователь попадал не на лендинг, а в рабочую часть;
- верхнеуровневая навигация была понятной и одинаково читаемой на desktop и mobile;
- `Поддержать` не терялось и оставалось заметным постоянным CTA;
- первое знакомство с интерфейсом было не формальным, а действительно объясняло продукт.

## Утверждённые продуктовые решения

### 1. Default route

- Гость на `/` видит лендинг
- Залогиненный пользователь на `/` редиректится в рабочий экран
- Рабочий экран по умолчанию: `Споры` / `/dashboard`
- Лендинг остаётся доступным по клику на логотип `Konsensus`

### 2. Верхний уровень навигации

Утверждённый набор верхнеуровневых разделов:
- `Споры`
- `Открытые`
- `Арена`
- `События`
- `Профиль`

Отдельный акцентный слой:
- `Поддержать`

### 3. Правило для `Поддержать`

`Поддержать` не прятать и не убирать из постоянного интерфейса.

Desktop:
- в верхнем shell — заметная кнопка `Поддержать`
- в нижней части экрана / footer-shell — яркие CTA `Boosty` и `Crypto`

Mobile:
- рабочая навигация должна быть постоянной
- support-strip не должен быть навязчивым рекламным баннером на каждом экране
- быстрый доступ к поддержке остаётся в `Профиле`, `/support` и в самом лендинге

### 4. Mobile navigation

Текущий hamburger/accordion не считать финальным паттерном для основного входа.

Целевой паттерн v1:
- fixed bottom navigation для top-level разделов
- support-strip над bottom navigation
- одинаковая IA с desktop, но другой контейнер

### 5. Лента / события

Текущая `Лента` не должна оставаться пустым экраном со слабой ролью.

Целевой смысл раздела:
- это `События`, а не просто список публичных споров
- туда входят:
  - релизы
  - активные публичные споры
  - события арены
  - позже community activity

До пересборки feed этот раздел не использовать как default home.

### 6. First-run onboarding

Онбординг делать в два слоя:

1. `Welcome overlay`
- показывается один раз после первого входа
- крупная приветственная подача
- коротко объясняет смысл продукта и 5 основных разделов

2. `Contextual coach marks`
- отдельные подсказки на конкретных страницах
- показываются один раз на страницу
- объясняют конкретные действия на экране
- должны уметь запускаться повторно прямо из этого экрана, если пользователь вернулся позже и забыл логику

### 7. Визуальная подача onboarding

Утверждённый принцип:
- слева крупный текст
- справа живое превью раздела или функции
- onboarding должен выглядеть как важный интерактивный вход, а не как маленькая скучная подсказка
- стиль должен быть в визуальном языке главного экрана Konsensus, а не в виде стандартного tooltip-only тура

## Критерий готовности v1

- залогиненный пользователь больше не попадает на лендинг по умолчанию
- основные разделы читаются без объяснений из скрытого hamburger
- `Поддержать`, `Boosty`, `Crypto` остаются постоянно заметными
- first-run onboarding действительно объясняет продуктовую структуру
- `Споры` становится ясным default home

## Что пока не делаем

На первом этапе не делать:
- сложную персонализацию стартового экрана по поведению пользователя
- AI-подбор стартового маршрута
- полноценную social feed ranking system
- тяжёлую анимационную сцену на каждый шаг onboarding

## Почему это важно

Сейчас проект уже богат функциями, но product shell объясняет их слабее, чем сам продукт.

Этот блок нужен, чтобы:
- ускорить первое понимание интерфейса;
- убрать ощущение пустых или странно названных разделов;
- сделать поддержку проекта заметной, но не спутанной с рабочей навигацией;
- дать залогиненному пользователю нормальную рабочую точку входа.
