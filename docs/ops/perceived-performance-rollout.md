# Perceived Performance Rollout

## Цель

Сделать интерфейс Konsensus визуально быстрее без тяжёлой серверной перестройки:
- кнопки должны мгновенно показывать, что действие началось;
- переходы между экранами не должны выглядеть как пустая пауза;
- основной контент должен появляться раньше вторичных деталей.

## Статус блока

- Стадия: `implemented_v1`
- Текущий релиз: `Perceived Performance v1`
- Возвращаться к блоку через этот файл и обновлять в нём выполненные шаги до синхронизации `status/roadmap`

### Выполнено в v1

- Добавлены route-level loading states для основных экранов и flow-веток
- Для ключевых server-action форм добавлены явные pending states
- Основные пользовательские CTA перестали выглядеть "мёртвыми" в момент ожидания сервера
- Появился единый reusable loading-shell для основных экранов

## Что входит в v1

### 1. Pending states

Ключевые формы теперь обязаны:
- сразу менять текст кнопки;
- блокировать повторный submit;
- визуально показывать, что серверное действие уже началось.

### 2. Route-level loading

Для основных маршрутов добавлены `loading.tsx`, чтобы переходы не выглядели как остановка интерфейса:
- `dashboard`
- `feed`
- `matchmaking`
- `profile`
- `arena`
- `dispute`
- общий fallback loading

### 3. Приоритетная визуальная отрисовка

На уровне v1 это решается не переписыванием fetch-логики, а тем, что:
- skeleton появляется сразу;
- пользователь видит каркас нового экрана до полной догрузки данных;
- вторичные детали не блокируют ощущение перехода.

## Пакет реализации v1

### Основные файлы

- новые `loading.tsx` в `src/app/**`
- reusable loading-компонент в `src/components`
- формы и submit-кнопки на ключевых страницах и в ключевых компонентах

### Что меняем в коде

- ввести общий loading-shell для внутренних экранов;
- закрыть pending-state на основных server-action формах;
- не менять серверную архитектуру спорных флоу, если задача решается мгновенной обратной связью в UI;
- оставить более глубокую backend-оптимизацию на следующий виток.

## Что пока не делаем

На первом этапе не делать:
- очередь фоновых задач для email / Telegram / AI;
- выделенный job-processing слой;
- агрессивную оптимизацию всех SQL-запросов подряд;
- рефакторинг server actions только ради микровыигрыша в latency.

## Что нужно добить после v1
