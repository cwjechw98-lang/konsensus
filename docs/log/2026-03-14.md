# Лог разработки — 2026-03-14

## Сессия 2: Критические фиксы + AI coaching + Медиация

### Что сделано

#### Критические баги
- **Rate limiting** в `createDispute`: максимум 5 активных споров на пользователя
- **Валидация входных данных**: название минимум 3 символа, описание минимум 5 символов
- **AI error states**: `triggerMediation` больше не крашится если Groq недоступен — сохраняет fallback-сообщение и завершает медиацию

#### Принятие решений в медиации (`CRITICAL #1`)
- Добавлено server action `acceptSolution` — участники могут принять одно из предложенных ИИ решений
- Создан `MediationClient.tsx` — клиентский компонент с real-time обновлениями через Supabase Realtime
- Когда оба участника принимают одно решение → статус `"accepted"`, отображается баннер "Консенсус достигнут! 🎉"
- UI показывает кто уже принял какое решение в реальном времени

#### AI waiting insight (подсказки ИИ во время ожидания)
- **Новая таблица** `waiting_insights` (миграция `docs/migration-waiting-insights.sql`)
- Функция `generateWaitingInsight` в `ai.ts` — генерирует персональную подсказку для участника сразу после его отправки аргумента
- Подсказка объясняет ПОЧЕМУ оппонент думает иначе — coaching без суждений, 2-3 предложения
- Отображается в `RealtimeDisputeClient` в "waiting" состоянии под "Ждём ответа..."
- Исчезает автоматически когда оппонент ответил
- Real-time обновление: новая подсказка появляется через Supabase Realtime без перезагрузки страницы

### Миграции для запуска
```sql
-- docs/migration-waiting-insights.sql
-- Запустить в Supabase SQL Editor
```

### Решения
- `acceptSolution` использует `createAdminClient()` для обхода RLS (т.к. RLS для resolutions может не быть настроен)
- `generateWaitingInsight` вызывается синхронно в `submitArgument` — добавляет ~1-2 сек к отправке аргумента, приемлемо для MVP
- `existingArgs` расширен до `author_id, round, position, reasoning, evidence` для контекста ИИ-подсказки

## Сессия 4: ИИ-углубление + пост-конфликтный момент

### Что сделано

#### Публичная карточка ИИ после раунда (`round_public_summaries`)
- Новая таблица `round_public_summaries` (миграция 00007)
- `generatePublicRoundSummary()` в `ai.ts` — нейтральное наблюдение, видимое обоим участникам
- Convergence score -2..+2 (↘↘ расхождение → ↗↗ сближение), цветовая кодировка
- Realtime-обновление через `RealtimeDisputeClient`
- Отображается после каждого завершённого раунда, выше приватного инсайта

#### Оценка силы аргумента перед отправкой
- `evaluateArgument()` server action — возвращает score 1–5, strengths[], suggestion
- `ArgueFormClient.tsx` — клиентская форма с двухшаговым flow: заполнить → "✨ Проверить" → увидеть оценку → отправить
- Карточка оценки: точки-индикатор, галочки сильных сторон, совет 💡
- Покрывает оба layout-а: round 1 (полная форма) и round 2+ (чат-стиль)
- Graceful degradation: если Groq недоступен — форма работает как раньше

#### Пост-конфликтный момент (ConsensusCelebration)
- Заменяет простой баннер "Консенсус достигнут!" на полноценный celebration screen
- Конфетти: 72 частицы, 8 цветов, pure CSS (confettiFall + confettiDrift keyframes), без external libs
- Анимированный баннер с gradient text, glow-пульс, star-pop
- Секция "Что вас объединяет" — common_ground из AI-анализа
- Инлайн-карточка достижения "Консенсус" (+30 очков) с floating points анимацией
- One-time подсказка "Отслеживайте достижения в ЛК" через localStorage (dismissable)

#### Документация
- `docs/roadmap.md` — полная актуализация: фазы 0–10, отмечены все выполненные пункты
- `docs/status.md` — полная перезапись: актуальный стек, схема БД, ключевые компоненты
- `CLAUDE.md` — обновлён стек (Groq вместо Claude API), секция "Текущий статус"

### Архитектурные решения
- Конфетти — seededRandom(42) для SSR-совместимости (детерминированные данные)
- `ArgueFormClient` сохраняет совместимость: `<form action={submitArgument}>` работает из клиентского компонента
- `ConsensusCelebration` живёт в `MediationClient` — клиентский компонент, реагирует на Realtime
