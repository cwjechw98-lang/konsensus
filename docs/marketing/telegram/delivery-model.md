# Telegram Delivery Model

## Желаемая модель

Пользователь хочет такую схему:
- в канал или группу уходит полный пост;
- в бот уходит только короткое уведомление о новом посте;
- если пользователь уже подписан на канал или группу, бот не дублирует этот анонс.

## Что есть сейчас

Базовая модель второго слоя уже внедрена:
- канал или группа получают полный structured release card;
- бот получает только короткий teaser;
- для bot-teaser есть suppress-логика по membership в целевом канале;
- membership-check пишется в SQL-кэш `telegram_channel_memberships`.

## Что технически нужно для правильной реализации

### 1. Целевой chat/channel id

Нужно явно хранить id канала или группы, по которой проверяется подписка.

### 2. Проверка подписки

Telegram Bot API даёт метод `getChatMember`, но в документации указано: он гарантированно работает для других пользователей только если бот администратор в чате.

Практический вывод:
- бот должен быть администратором канала или группы;
- без этого проверка членства будет ненадёжной.

### 3. Логика рассылки

Нужна новая ветка доставки:
- канал получает полный пост;
- бот получает только тизер;
- перед bot-уведомлением проверяется членство пользователя в канале/группе;
- если пользователь уже подписан, bot-уведомление пропускается.

### 4. Состояние и кэш

Кэш уже вынесен в отдельную таблицу:
- `telegram_channel_memberships`

Он обновляется:
- при release-send через `getChatMember`;
- через webhook по `chat_member` / `my_chat_member`, если бот получает такие апдейты.

## Почему не стоит смешивать это с текущим release-flow без доработки

Потому что сейчас release-flow решает другую задачу:
- быстрый UX-release;
- один payload;
- отправка в bot/channel/both.

Редакционная модель `канал = полный пост`, `бот = тизер + suppression` — это отдельный продуктовый слой.

## Что ещё не закрыто

Этот пакет не решает:
- штатное расписание публикаций;
- delivery analytics/reporting;
- richer editorial branching по типам постов.
